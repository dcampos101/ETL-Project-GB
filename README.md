# Data Engineering Challenge - PoC Data Pipeline

This project consists of a Proof of Concept (PoC) for a robust Data Pipeline and API service developed for **Globant**. The solution covers bulk historical data ingestion, real-time transaction processing, backup/restore management, and the exposure of analytical business metrics integrated with BI tools.

---

## Solution Architecture (Google Cloud Platform)

The architecture follows the principles of separating transactional and analytical responsibilities using native **GCP** services:

*   **Ingestion & Orchestration**: 
    *   **Cloud Data Fusion**: Acts as the main ETL pipeline engine for the bulk historical data load. It extracts the raw CSV files from Cloud Storage, applies data cleaning directives via Wrangler, and loads them into BigQuery efficiently.
    *   **Python & FastAPI**: Used to expose asynchronous, robust endpoints with native data validation via **Pydantic** for real-time transactions, backup, and restore services.
*   **Object Storage**: **Google Cloud Storage (Buckets)** is used as the staging data lake to store the initial CSV files and the backups generated by the backup service.
*   **Transactional Database**: **Cloud SQL** acts as the consistency receiver for initial transactional system processing.
*   **Data Warehouse & Analytics**: **BigQuery** centralizes all massive analytical storage for historical data. Optimized **SQL Views** were implemented to delegate heavy computing to the cloud engine.
*   **Data Visualization**: **Looker Studio (formerly Data Studio)** connects directly to the BigQuery SQL Views to render real-time interactive business dashboards without saturating API resources.

##  API Endpoints

The API includes built-in interactive documentation automatically generated at the `/docs` path (Swagger UI).

###  1. Ingestion & Data (POST)
*   `POST /historical/load_historic`: (Optional/Extension) Triggers the Cloud Data Fusion pipeline execution via REST API to start the historical bulk load from the storage bucket to BigQuery.
*   `POST /transactionality/process_trans`: Receives a JSON payload containing single or batch new transactions, validating data types before hitting the database destination.

###  2. Maintenance (POST)
*   `POST /backup/process_backup`: Generates a structured snapshot of the current tables and stores it securely in a Cloud Storage bucket.
*   `POST /restore/process_restore`: Fetches a specific backup file from the Cloud Storage bucket and restores the state of the tables.

###  3. Reports & Metrics (GET)
*   `GET /reports/emp_metrics_hires`: Returns the number of employees hired for each job and department in 2021, divided by quarter (Q1, Q2, Q3, Q4), ordered alphabetically.
*   `GET /reports/emp_metrics_hires`: Lists the IDs, names, and number of employees of departments that hired more personal than the global average of 2021, ordered descending.

##  Requirements & Installation

### Prerequisites
1. **Python 3.9+** installed.
2. A Google Cloud Platform account with a **Service Account** holding `BigQuery Admin`, `Storage Admin`, and `Cloud SQL Client` roles.
3. The Service Account key downloaded in JSON format.

### Local Execution Steps

1. **Clone the repository:**
   ```bash
   git clone https://github.com/dcampos101/ETL-Project-GB.git
   cd your-repository
   ```

2. **Configure environment variables:**
   Set the path to your Google Cloud credentials:
   ```bash
   export GOOGLE_APPLICATION_CREDENTIALS="absolute/path/to/your/service_account_key.json"
   ```

3. **Install dependencies:**
   ```bash
   pip install -r src/requirements.txt
   ```
   *(Make sure to include `fastapi`, `uvicorn`, `pydantic`, `google-cloud-bigquery`, `google-cloud-storage`, `pandas`, and `pyodbc/sqlalchemy` in your dependencies).*

4. **Run the API server:**
   ```bash
   uvicorn src.transactionality.main:app --reload
   ```
   The API will be up and running interactively at: `http://127.0.0`

##  Design Decisions & Technical Seniority

*   **No-Code/Low-Code Scale with Cloud Data Fusion**: Instead of writing complex, hard-to-maintain custom Python scripts for heavy historical data, **Cloud Data Fusion (CDAP)** was chosen. Its **Wrangler** component handles schema parsing, character encoding, and quoted-value alignment visually, translating the logic into optimized Apache Spark jobs under the hood to load BigQuery seamlessly.
*   **Early Validation for Streaming**: Pydantic is used in the FastAPI request schemas to guarantee that no corrupt data (wrong data types or unexpected nulls) reaches Cloud SQL or BigQuery during real-time streaming, responding immediately with standard HTTP 422 errors.
*   **Analytical Compute Pushdown to Looker Studio**: Complex metrics queries (quarterly pivots and dynamic moving average calculations) are performed via **BigQuery Views** instead of being computed at the application level. This avoids transferring millions of rows over the network, minimizing local RAM usage and allowing **Looker Studio** to consume pre-aggregated data directly with sub-second dashboard rendering times.
